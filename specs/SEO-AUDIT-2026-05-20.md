## Critical findings (real SEO impact)

### F1 · 32 brief permalinks have thin descriptions (<35 chars)

**Pages:** the Feb / early-Mar brief permalinks where `telegramText` is empty in the archive. Per memory, these are "unfixable in landing scope" — content was never captured.

**Current state:** descriptions like `"MARKET WRAP · February 23, 2026: "` — cut off after the colon, no body to extract.

**Risk:** Google treats thin-content URLs as a quality signal against the WHOLE brief permalink namespace. 32 pages out of 331 is ~10%. Enough to affect domain-level quality scoring.

**Fix options (recommend B):**
- **A.** Drop them from sitemap (don't ask Google to crawl). Pages still exist for direct links.
- **B.** Add `<meta name="robots" content="noindex,follow">` to brief permalinks where `telegramText` is empty. Explicit "don't index this thin page" signal.
- **C.** Generate a fallback description like "AgentCanary {SLOT} brief archive entry for {DATE}. Brief content unavailable for this date in the archive." (Still thin, just explicit.)

**Effort:** ~30 min. One conditional in `renderBrief`.

### F2 · 75 asset pages share the homepage OG image

**Current state:** every `/assets/{TICKER}/` page returns `og:image = https://agentcanary.ai/og-image.png`. Sharing /assets/BTC/ on X/LinkedIn/Discord shows the generic AC card, not a BTC-specific one.

**Risk:** Lower social CTR (image doesn't match link content) + lost branding opportunity (we already generate per-brief OG cards — same primitive could apply to assets).

**Fix:** generate a per-asset OG card via existing `lib/og-card.mjs` + `@resvg/resvg-js` pipeline. Lead with ticker + hit rate from the per-asset data we have. ~2h of work for the 75-card generator + 1-line render change.

**Effort:** half-day.

### F3 · /sources/ page has zero JSON-LD

**Current state:** `/sources/index.html` is hand-written, doesn't go through `wrapPage`, has no structured data at all. Other pages have 1-4 JSON-LD blocks.

**Risk:** Missed schema.org parsing. We document 28 datasets there — each is a `Dataset` entity that could surface in Dataset Search if marked up. Also missed AggregateRating, since we have the public Brier number.

**Fix:** add `<script type="application/ld+json">` with either (a) a DataCatalog mirroring homepage's, or (b) a Reference / TechArticle type with itemListElement[] of 28 named datasets.

**Effort:** ~1h. Mostly content authorship (which dataset gets which Schema type).

## Medium-priority findings

### F4 · Asset + regime page descriptions stuck at ~75 chars

**Current state:**
- `/assets/{TICKER}/`: "Every AgentCanary brief that mentioned X. N mentions, time-sorted desc." (73-75 chars typical)
- `/regimes/{slug}/`: "Every AgentCanary brief published during the X regime. N briefs, time-sorted desc." (~85 chars)

Google's snippet space is ~155-160 chars. We're using ~half.

**What's missing:** the proof-point. After today's shipping, we have:
- Per-asset hit rate / Brier score (in score-aggregates.json)
- Per-regime brief count + density

**Better:**
- `/assets/SPY/`: "SPY mentioned in 125 AgentCanary briefs (since 2026-02-23). 8.8% hit rate across scenario targets. Most-recent-first archive with permalinks."  → 145 chars
- `/regimes/overheating/`: "81 AgentCanary briefs published while macro regime was OVERHEATING. Regime call accuracy 80% over the period. Most-recent-first."  → 132 chars

**Effort:** ~30 min. `renderAsset` + `renderRegime` get hit-rate from score-aggregates.json + numerics from mentions array.

### F5 · Hand-written pages bypass wrapPage — drift risk

**Pages:** `index.html`, `sources/index.html`.

**Drift evidence from this session:**
- Generic page-level fixes (e.g. the SEO footer page-aware change) automatically applied to 500+ pages via wrapPage. Hand-written pages had to be touched separately.
- index.html `<head>` got llms-full + describedby links via separate patch. Sources/index.html still has only its hand-written head.

**Risk:** every future site-wide tag change requires remembering to update hand-written pages too. Mistakes compound.

**Fix options:**
- **A.** Bring hand-written pages into wrapPage (operator-driven design refactor — non-trivial because the homepage has bespoke layout).
- **B.** Extract `<head>` block into a shared `_renderHead(opts)` helper used by both wrapPage AND the hand-written pages. Keeps custom layouts but standardizes head.
- **C.** Accept the drift, add a build-time check (e.g. assert `llms-full.txt` link present in every HTML) that catches drift before deploy.

**Effort:** A = 1-2 days, B = 4h, C = 30 min. Recommend C (cheap, catches future drift, doesn't change current rendering).

### F6 · No pre-build SEO regression test

**Observation:** the dupe-snippet bug, drift on tool counts (17/18/19), and the now-32-thin-descriptions weren't caught by any automated check.

**Lean test idea (~1h):** add `tools/seo-check.mjs` that runs after `build-record-v1.mjs`:
- Asserts every generated HTML has: title (10-65 chars), description (40-160 chars), canonical, og:image, og:title, og:description, h1 count = 1.
- Asserts description uniqueness above a threshold (e.g. >95% of pages have unique descriptions).
- Exits 1 + lists violations.

**Effort:** ~1h. Net: future drift gets caught before push.
